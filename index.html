Problem

All students above first year at the University of Cape Town (UCT) are eligible to park on campus if they buy a parking disk. Due to a steady increase in the number of students attending UCT, the number of students per bay has increased dramatically. This has lead to many students driving around campus for hours in search of a parking bay. The Properties and Services department is seeking a solution to this inconvenience.

Solution

We have designed a mobile solution, ParkCheck, to solve this problem on upper campus. The Properties and Services department has agreed to provide the number of free bays per student parking area every minute. This allows us to determine the current parking availability, and predict future availability based on past data. ParkCheck visualizes the availability of parking per parking lot currently, and shows the predicted availability throughout the day.

Visual Queries

1. Where can I park right now?

Students need to be able to quickly see where they can park right now. If we can effectively answer this question, students won't have to spend valuable time driving around in search of parking. This is of the highest importance, as our student users are most likely to use the app to find parking right now.

2. Where can I park later?

If students can see the predicted parking availability for the day, they can make an educated decision on when to drive to campus. This compliments query 1, as it allows proactive students to go when the parking availability is high, potentially alleviating some of the pressure during peak hours.

3. Where is each parking area?

Campus can be confusing to navigate and people usually like to get parking bays closest to their destination. However, with time, students become familiar with the roads and locations of parking areas. Also, all parking areas are within walking distance of the whole of upper campus. This is why this query is ranked third.

A literature review into existing parking applications was conducted to identify the advantages and disadvantages of current implementations. These were analysed and combined with the feedback received on the initial design presentation to yield the final design. The most similar current applications identified were ParkMe, SpotHero and Parker. All of the current implementations found were involved a map application that allowed for scrolling, zooming and current location look-up. Overlaying the parking information over the 2-dimensional data of parking location proved to be very intuitive due to the familiarity of map based applications. The zoom support also helps to alleviate the effect of the small screen size constraint by allowing users to manipulate the map and data to see an overview or to focus in on specific areas. Most current applications utilized a colour gradient to help visualize the relative parking costs. Only one application, Parker, used a colour gradient to depict parking availability.All of the applications used markers and symbols to depict the parking locations. Most of the apps used the same symbol for all parking lots, and just varied the colour as per the colour gradient. Only Parker used different symbols to differentiate between street parking and parking lots. This approach allowed for faster visual queries relating to looking for different types of parking. All of the applications allowed for more information to be made visible once a parking was clicked. This allowed for a streamline overview that was not cluttered by information while still being comprehensive.

ParkMe

ParkMe is a mobile application that focuses on comparing parking lots by by price and distance to destination. It displays the exact distance from each parking facility to the selected destination, and how much it will cost. These are 2 common queries drivers have when looking for parking in a city, but are less applicable to the UCT use-case. On UCT upper-campus all parking lots are within walking distance from all venues, and all student bays have the same pre-paid fee. ParkMe makes good use of an intuitive colour gradient on its markers - the red marked parking areas showing less optimal parking opportunities than orange or green. We thought this was useful in quickly identifying cheap parking lots. We used this in our designs to depict price, not availability.

Iteration 1

This initial attempt tries to utilize a list view of the different parking lots. The outside border of each item correlates to a green to red availibility gradient. Each item attempts to visualize the parking data in a different way so as to most effictivly transfer the intended query information. This attempt lacked efficacy in many ways including: parking location, obvious meaning for all components and support for visual queries. Therefore other approaches were investigated.

Iteration 2

This attempt improves on the initial attempt by adding a map of UCT to attempt to support the "Where is the parking lot" query." It uses a green (empty) to red (full) colour gradient to inform users as to the number of spaces availible. It also uses the same gradient as a lookup connection between the list view and the spatial parking lot locations. The list view attempts to rank the parking lots in decending order of space availible. This attempt neglected the time based component of the parking data and was also slightly confusing as to which list item correlated to which parking lot on the map.

Iteration 3

This design attempted to improve on the last design by re-introducing the temporal parking information into the visualization. This design still implements a list view but now connects the list view to the spatial parking lot using the number of free bays as a symbol. This design lacked in intuitivity and is not particularly effective at conveying its intended meaning.

Design Feedback

  1. More visual queries. Queries such as parking closest to certain buildings or at certain times on other days.
  2. A useful colour gradient along with texture and symbols could be used to assist visual queries.
  3. Number of bays left as a percentage of the maximum as well as the number of bays could be useful.
  4. Implement a highlight for the best parking lot right now.
  5. Show all values as number of bays remaining as opposed to number of filled bays.

Applying Feedback

It was decided that more research was needed to investigate the effective use of colour gradients as well as time varying complex data. The illustration (left) from google maps helped in this area. The colour gradients used by google maps to show the different types of traffic vary in intensity. This aids in their differentiation for colourblind users. This variation on two channels makes for a very distinct difference between the gradient values. An identical gradient was selected for the final ParkCheck design to indicate amount of parking available. The slider bar for time variation is simple and easily understood. It was decided that a simplified version of the time slider, with added emphasis on the slider position relating to current conditions would be added to the final design. A button would be added to allow for easy return to current conditions. The final application would be implemented on a fully functioning map to benefit from the map manipulation capabilities mentioned in the literature review. The number of free bays would be displayed on the parking lot whilst in overview, and would vary along with the stored data when the slider value is changed. On-click events that bring up small windows that occlude part os the map detailing more information on each parking lot such as any restrictions, number of free bays vs total bays and the full name of the parking lot will be implemented. It was decided that implementing a "highlight" function that would inform a user of the best parking lot was not necessary as this would add clutter over and above all the standard information which is already difficult to incorporate due to screen size restrictions.

Advantages

The final implementation boasts a simple and sleek overview design that allows for access to the primary visual query: "Where to park now?". It does this using an easily identifiable colour gradient as well as user's inate map based app usage fimiliarity. The final design manages to avoid clutter, while still clearly and easily supporting time based queries. The map based implementation of the final app leads to an easy to use interface that somwhat circumvents the screen size constraints. On-click pop-ups give extra information without cluttering or occluding data in overview mode.

Disadvantages

Data in the on-click pop-ups is not immediately availible. The final design also somewhat hides the temporal variation of the parking availibility. This was deemed acceptable as the temporal data is not critical to the primary visual query, but is used in the secondary query only.
